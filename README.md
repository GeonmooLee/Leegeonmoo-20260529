# 살뜰: AI 구매 실행 도우미

살뜰은 가족 일정과 생활용품 구매가 뒤섞인 메모에서 살 물건을 정리하고, 필요한 시점에 상품 후보를 찾도록 돕는 풀스택 MVP입니다. 식재료 구매 앱이 아니라 소풍 준비물, 생일 선물, 새 학기 준비물, 가족 생활용품처럼 일정에서 생기는 구매 과제의 기억 부담을 줄이는 실행 도구입니다.

## 주요 흐름

1. 사용자가 메모함에 가족 메모를 여러 개 저장합니다. 메모 저장만으로 AI API를 호출하지 않습니다.
2. 새 메모의 `AI 분석하기`를 눌러 저장과 분석을 바로 진행하거나, 저장된 메모 카드의 `AI로 분석하기` 또는 `다시 분석` 버튼을 누릅니다.
3. 해당 메모에 종속된 분석 작업 화면에서 AI가 일정별 물품을 `사용자 언급 물품`과 `AI 추가 추천 물품`으로 분리하고, 맥락에 맞는 `AI 권장 액션 시점`을 참고용으로 제안합니다.
4. 사용자는 분석 결과를 수정하거나 직접 항목을 보완한 뒤 영구 쇼핑 리스트에 담습니다.
5. 쇼핑 리스트에서 여러 메모의 물품을 함께 관리하고 직접 찾은 `상품 링크 등록` 또는 `AI로 상품 찾기`를 실행합니다.
6. 사용자가 쇼핑 리스트에서 `AI로 상품 찾기`를 누른 경우에만 서버가 공개 상품 검색 결과를 수집하고, AI가 일정 맥락과 배송 정보 유무를 고려해 최대 3개 후보를 추천합니다.
7. 사용자는 상품을 선택하고 실제 구매가 끝나면 `구매 완료`로 표시합니다.

## 구현

- Frontend: React, Vite, JavaScript
- Backend: Node.js, Express, JavaScript
- AI: OpenAI Responses API의 JSON Schema 기반 구조화 출력
- Product search: 네이버 쇼핑 검색 API를 메인 상품 후보 수집 방식으로 사용
- Crawling: Cheerio를 이용한 상품 링크의 공개 메타데이터 경량 수집(lightweight crawling)
- Persistence: 브라우저 `localStorage`
- Deployment: Vite 빌드 결과를 Express가 정적 파일로 제공

브라우저에는 여러 메모, 메모별 AI 분석 결과, 확정한 쇼핑 리스트, 구매 완료 기록, 선택한 상품 링크가 저장됩니다. 로그인은 필요하지 않습니다.

주요 `localStorage` 키:

- `salddeut.memos`: 여러 메모 목록
- `salddeut.analysesByMemoId`: `memoId`로 연결된 분석 결과
- `salddeut.shoppingList`: 여러 메모에서 확정한 영구 쇼핑 리스트

## 페이지 구조

- `/`: 소개와 `지금 시작하기` 버튼이 있는 홈
- `/memos`: 여러 메모를 작성하고 관리하는 메모함
- `/memos/:memoId`: 저장한 메모의 전체 내용과 연결된 물품을 확인하는 상세 화면
- `/memos/:memoId/analysis`: 특정 메모에 종속된 AI 분석 작업 화면
- `/shopping-list`: 모든 메모에서 추가한 물품이 모이는 쇼핑 리스트

각 화면은 독립 URL을 사용합니다. 새로고침, 직접 URL 접근, 브라우저 뒤로가기를 지원합니다.

## 로컬 실행

Node.js 20 이상을 권장합니다.

```bash
npm install
copy .env.example .env
npm run dev
```

개발 중 프론트엔드는 `http://localhost:5173`, API 서버는 `http://localhost:3001`에서 실행됩니다. Vite 개발 서버가 `/api` 요청을 Express로 전달합니다.

API 키가 없어도 화면 흐름을 확인할 수 있습니다. 이 경우 예시 메모 분석과 수집된 상품 기준의 데모 추천을 사용하며, UI에 데모 모드임을 표시합니다.

## 환경 변수

`.env.example`을 `.env`로 복사하고 값을 설정합니다.

```dotenv
OPENAI_API_KEY=
OPENAI_MODEL=gpt-5.4-mini
OPENAI_RECOMMEND_MODEL=gpt-5.5
NAVER_CLIENT_ID=
NAVER_CLIENT_SECRET=
PORT=3001
```

- `OPENAI_API_KEY`: 실제 AI 분석과 AI 상품 추천에 필요합니다.
- `OPENAI_MODEL`: 메모 구매 항목 추출, 카테고리 분류, 중복 정리 등 기본 분석에 사용하는 모델입니다. 비용이 낮은 모델(예: `gpt-5.4-mini`)을 권장합니다.
- `OPENAI_RECOMMEND_MODEL`: `AI로 상품 찾기`의 검색어 정제, 사용자 상황 기반 추천 이유 작성, 여러 상품 비교 후 최적 후보 선정에 사용하는 모델입니다. 추론 품질이 더 높은 모델(예: `gpt-5.5`)을 권장합니다. 설정하지 않으면 `OPENAI_MODEL` 값을 그대로 사용합니다.
- `NAVER_CLIENT_ID`, `NAVER_CLIENT_SECRET`: `AI로 상품 찾기`에서 네이버 쇼핑 검색 API를 호출하는 데 필요합니다.
- `PORT`: Express 서버 포트입니다.

`AI로 상품 찾기`에서 네이버 쇼핑 검색 API를 사용하려면 [네이버 개발자센터](https://developers.naver.com/apps/#/register)에서 애플리케이션을 등록하고 `검색` API 사용을 추가해 `Client ID`와 `Client Secret`을 발급받아야 합니다. 발급받은 값을 `NAVER_CLIENT_ID`, `NAVER_CLIENT_SECRET`에 설정해 주세요. 네이버 키와 OpenAI 키는 모두 백엔드 환경변수로만 사용하며, 프론트엔드에는 절대 노출되지 않습니다.

## 배포

```bash
npm install
npm run build
npm start
```

Express가 `dist` 폴더를 제공하므로 Node.js를 실행할 수 있는 서비스에 그대로 배포할 수 있습니다. 배포 환경에 `OPENAI_API_KEY`, `OPENAI_MODEL`, `OPENAI_RECOMMEND_MODEL`, `NAVER_CLIENT_ID`, `NAVER_CLIENT_SECRET`, `PORT`를 설정해 주세요.

## AI로 상품 찾기 동작 방식

`AI로 상품 찾기`는 쇼핑 리스트에서 사용자가 직접 버튼을 누른 경우에만 실행됩니다. 초기 메모 분석 단계에서는 상품 검색이나 크롤링을 하지 않습니다. 전체 흐름은 `POST /api/products/recommend` 한 곳에서 처리하며, 다음 순서로 동작합니다.

1. 선택한 물품의 맥락(물품명, 관련 일정, 필요 날짜, AI 권장 액션 시점, 권장 이유, 비고, 사용자 의도)을 백엔드로 전달합니다.
2. OpenAI가 맥락을 바탕으로 정제된 검색어(refinedSearchQueries)와 추천 기준(recommendationCriteria, avoidCriteria)을 생성합니다. 물품 범위가 너무 넓으면 `needsMoreInfo=true`를 반환하고, 프론트에서 예산·대상·선호·피하고 싶은 조건을 입력받아 다시 시도합니다.
3. 네이버 쇼핑 검색 API로 상품 후보를 약 10개 수집합니다. 결과가 적으면 두 번째 검색어로 보완합니다. 상품명의 HTML 태그는 제거하고 가격은 숫자로 정규화합니다.
4. 최종 추천(또는 폴백 상위 3개) 상품 링크에 대해서만 가벼운 정적 lightweight crawling을 시도해 공개 메타데이터(pageTitle, description, og 태그 등)를 수집합니다. 전체 HTML은 OpenAI에 보내지 않습니다.
5. 제목·링크 누락, 중복, 핵심 키워드 불일치 후보를 거르는 기본 필터링을 거칩니다. 후보가 3개 미만이면 필터를 완화하고 UI에 안내를 표시합니다.
6. OpenAI가 수집된 후보 안에서만 최대 3개를 추천합니다. 상상으로 새로운 상품을 만들지 않으며, 추천 상품의 제목·가격·판매처·이미지·배송 정보는 수집된 후보 값으로 강제 동기화합니다.

네이버 스마트스토어 등 대부분의 상품 상세 페이지는 배송 정보를 JS로 렌더링하고 서버 측 요청을 적극적으로 차단(예: HTTP 429)하므로, 정적 lightweight crawling으로는 배송 정보를 얻지 못하는 경우가 많습니다. 이때도 추천 기능은 멈추지 않고 계속 진행하며, 배송 정보는 추측하지 않고 `배송 정보는 상세 페이지에서 확인해 주세요.`라고 안내해 사용자가 상품 링크에서 직접 확인하도록 합니다. 헤드리스 브라우저 기반 배송 정보 자동 수집은 차단 문제로 이번 MVP 범위에서 제외했습니다.

### 키가 없거나 호출이 실패할 때

- 네이버 API 키가 없으면 예시 상품을 보여 주고 `네이버 쇼핑 API 키가 없어 예시 상품을 보여드리고 있어요.`라고 표시합니다.
- 네이버 API 호출이 실패하면 앱이 깨지지 않게 처리하고 `상품 후보를 불러오지 못했어요. 잠시 후 다시 시도해 주세요.`를 표시합니다.
- 크롤링이 실패해도 추천은 계속 진행하고, 해당 상품의 배송 정보는 `배송 정보는 상세 페이지에서 확인해 주세요.`로 안내합니다.
- OpenAI 추천 생성이 실패하면 네이버 후보 리스트라도 보여 주고 `AI 추천 이유를 생성하지 못해 검색 결과를 먼저 보여드려요.`를 표시합니다.

## 설계 결정

- AI가 사용자의 보유 물품을 알 수 없으므로 `구매 필요`와 `확인 필요`로 임의 분류하지 않습니다.
- 직접 언급한 물품과 맥락상 제안하는 물품을 분리해 AI 추천의 근거를 투명하게 보여 줍니다.
- 쇼핑 리스트에는 사용자가 확인해 담은 항목만 노출합니다.
- 상품 추천은 인기순만 따르지 않고 일정, 필요 날짜, AI 권장 액션 시점, 권장 이유, 메모, 수집된 배송 정보 유무를 함께 고려합니다.
- 날짜가 불명확하면 AI가 ISO 날짜를 만들지 않고 `null`과 낮은 신뢰도를 반환하도록 지시합니다.
- AI 권장 액션 시점은 고정 카테고리 DB나 물품별 하드코딩 규칙이 아니라 메모 맥락, 일정, 물품 특성, 일반적인 준비 리스크를 참고하여 추론합니다.

## MVP 한계

- 실제 캘린더 연동이 없습니다.
- 카카오톡이나 텔레그램 알림이 없습니다.
- 학교 안내문 자동 수집 기능이 없습니다.
- 결제 또는 장바구니 연동이 없습니다.
- 배송 예정일은 보장되지 않습니다.
- AI 권장 액션 시점은 정확한 배송일 예측이 아니라, 사용자의 메모와 물품 특성을 바탕으로 한 참고용 제안입니다. 실제 배송일과 재고는 상품 추천 단계에서 크롤링된 정보가 있는 경우에만 보완되며, 확인할 수 없는 경우에는 임의로 추정하지 않습니다.
- UI에서는 구매, 후보 비교, 기존 물품 확인처럼 사용자가 해야 할 행동과 시점을 함께 보여 줍니다.
- 크롤링 결과에 배송 정보가 없으면 추측하지 않고 `배송 정보는 상세 페이지에서 확인해 주세요.`로 안내합니다.
- 상품 추천은 네이버 쇼핑 API와 제한적 크롤링 데이터를 기반으로 하며, 전체 시장 최적 추천이 아니라 제한된 후보 안에서의 추천입니다.
- 공개 페이지 HTML 구조 변경이나 로봇 차단 정책에 따라 크롤링 결과가 달라질 수 있습니다.
- 현재 MVP의 상품 후보 수집은 네이버 쇼핑 검색 API만 사용하며, 쿠팡·다나와 등 다른 쇼핑몰 직접 크롤링이나 리뷰 수집은 포함하지 않습니다.

## 향후 개선

- 가족별 일정과 캘린더 동기화
- 알림 채널 연동 및 액션 시점 리마인드
- 학교 안내문 이미지 및 PDF 분석
- 사용자별 선호 브랜드, 예산, 사이즈 저장
- 제휴 쇼핑 API를 이용한 안정적인 상품 데이터 수집
- 쿠팡, 다나와 등 추가 쇼핑몰 상품 후보 수집과 리뷰 데이터 활용
- 실제 배송 가능일 검증과 장바구니 연결

## 프로젝트 회고

### 문제 정의와 타겟 선정 의도

이 프로젝트에서 다루는 "가족 일정에서 생기는 구매 과제를 놓치는 문제"가 분명한 페인포인트인 것은 맞지만, 제가 떠올린 다른 페인포인트들(예: '광고성 후기' 문제 등)보다 압도적으로 크다고 생각하지는 않았습니다. 오히려 광고성 후기 같은 문제가 더 강한 페인포인트일 수 있습니다.

그럼에도 이 문제를 선택한 이유는 **해결 수단이 'AI 에이전트'로 정해져 있고, 주어진 시간이 이틀뿐**이라는 제약 때문입니다. 이 조건에서는 "문제 정의"가 중요하다는 것을 알면서도, 해결 방법이 고정되어 있으니 **AI 에이전트가 강점을 가질 수 있는 페인포인트**를 타겟으로 잡을 수밖에 없었습니다.

### 핵심 설계 의도: "굳이 우리 서비스를 써야 하는 이유"

가장 경계한 지점은 *"AI 에이전트라면서 그냥 ChatGPT를 쓰면 되는 것 아닌가?"* 라는 질문이었습니다. 그래서 개발 내내 **'ChatGPT를 쓰지 않고 굳이 우리 서비스를 쓰게 만들겠다'**는 한 가지 목표에 집중했고, 그 차별점을 만들기 위해 다음을 설계 원칙으로 삼았습니다.

- **단발성 답변이 아니라 실행 흐름을 남긴다**: 메모 → AI 분석 → 쇼핑 리스트 → 상품 찾기 → 구매 완료로 이어지는 과정을 영구 저장해, 한 번의 대화로 끝나지 않고 "언제 무엇을 해야 하는지"를 계속 추적하게 했습니다.
- **사용자 맥락을 추천에 반영한다**: 단순 검색이 아니라 관련 일정·필요 날짜·AI 권장 액션 시점·비고·사용자 의도를 종합해 검색어를 정제하고 추천 이유를 작성합니다.
- **AI가 모르는 것을 지어내지 않는다**: 보유 여부를 임의로 분류하지 않고, 모호한 날짜는 `null` + 낮은 신뢰도로 두며, 배송일·재고는 추측하지 않습니다. 추천도 수집된 후보 안에서만 하도록 강제합니다.

### 아쉬웠던 점

가장 아쉬웠던 부분은 **배송 정보를 추천 기준에 넣지 못한 것**입니다. 네이버 쇼핑 API로 상품 후보를 가져온 뒤 각 상품 링크에서 배송 정보를 크롤링하려 했지만, 네이버 측에서 서버 요청과 헤드리스 브라우저를 **봇으로 인식해 차단(HTTP 429 등)**했습니다. 헤드리스 브라우저(Playwright)까지 시도했으나 동일하게 막혀, 결국 lightweight 크롤링으로 되돌리고 배송 정보는 "상세 페이지에서 직접 확인" 안내로 대체했습니다. 배송 시점은 "언제 사야 하는가"라는 이 서비스의 핵심 가치와 직결되는 정보라 더 아쉬웠습니다.

### 더 개선하고 싶은 방향

- **알림 기반 리마인드**: 일정 기간마다 카카오톡·텔레그램 등으로 "지금이 살 시점"이라는 알림을 보내, 사용자가 앱을 켜지 않아도 챙길 수 있게 만들고 싶습니다. 이때 AI 에이전트의 가치가 가장 커진다고 봅니다.
- **도메인 확장**: 쇼핑에 한정하지 않고 가족 전체 일정·장보기 등 "기억해야 할 가정 내 실행 과제" 전반으로 확장할 수 있는 아이템이라고 생각합니다.
- **안정적인 상품·배송 데이터**: 차단 문제를 우회하기 위해 제휴 쇼핑 API나 공식 데이터 소스를 활용해 배송 정보를 추천 기준에 정식으로 포함시키는 것이 다음 우선순위입니다.
